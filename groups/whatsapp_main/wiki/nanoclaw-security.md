# NanoClaw — Security Model

## Core principle

Security through **true OS-level isolation**, not application-level permission checks. Agents run in actual Linux containers and can only access what's explicitly mounted.

## Trust levels

| Entity | Trust Level | Rationale |
|---|---|---|
| Main group | Trusted | Private self-chat, admin control |
| Non-main groups | Untrusted | Other users may be malicious |
| Container agents | Sandboxed | Isolated execution environment |
| Incoming messages | User input | Potential prompt injection risk |

## Security boundaries

### 1. Container isolation (primary boundary)

- **Process isolation** — container processes cannot affect the host
- **Filesystem isolation** — only explicitly-mounted directories visible
- **Non-root execution** — runs as `node` user (uid 1000)
- **Ephemeral** — `--rm` flag, fresh environment per invocation

### 2. Mount validation

All mounts validated against `~/.config/nanoclaw/mount-allowlist.json`:
- Symlinks resolved before check
- Blocked patterns rejected (`.ssh`, `.env`, private key files, etc.)
- `nonMainReadOnly` setting forces read-only for non-main group mounts
- Relative `containerPath` required (no leading `/`); prefixed with `/workspace/extra/`

### 3. IPC authorisation

IPC operations validated by host before execution:
- Non-main groups can only `send_message` to their own chat
- Group registration (`register_group`) and `refresh_groups` are main-group-only
- Task scheduling checked against group privilege level

### 4. Sender allowlist

`~/.config/nanoclaw/sender-allowlist.json` controls which senders can trigger agent responses per group. Per-group configuration. Only listed senders can activate the agent.

**Note:** This file lives outside the container — changes take effect on next message check but do not persist across container host reboots unless the file is on the host filesystem (not inside the container).

Our allowlist includes Adam's WhatsApp sender (`61413403033@s.whatsapp.net`) for all teams.

### 5. Credential isolation via OneCLI Agent Vault

**Real API credentials never enter containers.** This is non-negotiable.

How it works:
1. Credentials registered once with `onecli secrets create`
2. When NanoClaw spawns a container, it calls `applyContainerConfig()` to route outbound HTTPS through the OneCLI gateway
3. Gateway matches requests by host/path, injects the real credential, and forwards
4. Agents cannot discover real credentials — not in environment, stdin, files, or `/proc`

**Per-agent OneCLI policies:** Each NanoClaw group gets its own OneCLI agent identity, allowing different credential scopes per group.

**What stays in `.env` (not vaulted):** Channel tokens (Telegram, Slack, Discord) remain in `.env` since they're not container-facing.

### 6. Host-mounted secrets (for file-based credentials)

For credentials that must be file-based (e.g. Google service account JSON):
- File stored on host at `~/nanoclaw-secrets/{team}/`
- Mounted read-only via `containerConfig.additionalMounts` in `registered_groups` DB
- Container reads from `/workspace/extra/{path}`, never writes back

Example (Insights Team google service account):
```json
{
  "additionalMounts": [
    {
      "hostPath": "~/nanoclaw-secrets/insights",
      "containerPath": "insights-creds",
      "readonly": true
    }
  ]
}
```

## What NEVER goes in version control or workspace files

- API keys of any kind
- OAuth tokens
- Service account JSON files
- Application passwords
- Webhook secrets

Use OneCLI vault (for API calls) or host-mounted secrets (for file-based auth). Document the *path* to secrets; never the secrets themselves.

## Credential patterns by type

| Credential type | Storage method | Access in container |
|---|---|---|
| Anthropic API key | OneCLI vault (type: `anthropic`) | Injected at gateway level |
| REST API keys (Bearer/Ghost/etc.) | OneCLI vault (type: `generic`, `valueFormat`) | Injected as header |
| Google service account JSON | Host file mount | Read from `/workspace/extra/insights-creds/google-service-account.json` |
| Ghost Admin API key | OneCLI vault (valueFormat: `Ghost {value}`) | JWT generated by agent using the injected key |
| WordPress Application Password | Host mount (in our setup) | Read from `/workspace/extra/wordpress-creds/` |

## Ghost Admin API auth note

Ghost Admin API requires JWT auth. Do NOT use `Bearer {value}` format:
```
Authorization: Ghost <token>
```
Token generated from `id:secret` key pair using `jsonwebtoken` library (HS256, 5-min expiry). The `valueFormat` in OneCLI must be `Ghost {value}`, not `Bearer {value}`.
